The Playbook

Four Automation Stages

The Comms Engine covers the entire patient communication lifecycle — from the moment a lead appears to reactivating patients who've disappeared.

Stage 1

Lead Response

Every inbound lead — call, form fill, or missed call — gets an intelligent response within 60 seconds, 24/7. AI voice answers overflow calls. Instant text-back for missed calls. Automated nurture if not immediately booked.

Cadence Immediate response → 4hr follow-up → Day 1, 3, 7 nurture sequence
Stage 2

Treatment Follow-Up

When treatment is presented and not accepted, an automated follow-up sequence activates. Messages are personalized, come "from" the provider, and use proven cadences that close 50%+ more treatment.

Cadence Day 3 (check-in) → Day 7 (education) → Day 14 (soft close) → Day 30 (final touch) → $20K+ cases: 5+ touches with voice
Stage 3

Rebooking

Patients who leave without booking their next appointment get same-day recovery outreach. Hygiene recall sequences activate at 5 months. No patient falls through the cracks.

Cadence 30 min post-visit → Day 1, 3, 7 recovery → 5mo, 5.5mo, 6mo recall
Stage 4

Database Reactivation

Monthly scans identify dormant patients (12+ months). Tiered outreach by patient value: AI voice calls for high-value, text sequences for standard, email for bulk. 130K+ dormant patients across the network.

Cadence Monthly scan → Tiered outreach (voice/text/email) → 3-touch over 30 days → Quarterly thereafter

Architecture

System Architecture

Five layers: Data Integration, Comms Engine Core, Communication Transport, Monitoring & Learning, and the minimal Human Interface.

Components

Core Components

Six primary components that power the Comms Engine — from the per-practice agent instances to the network-wide learning engine.

Practice Agent Instance

One autonomous agent per practice. Maintains practice context, executes all 4 playbook stages, personalizes using LLM.

  • Maintains provider roster, specialties, schedule patterns
  • Operates in Supervised, Monitored, or Autonomous mode
  • Escalates to human when confidence is low
  • Extends Practice Concierge agent pattern

Playbook Engine

The rule engine executing the SGA Patient Communication Protocol. Manages cadences, triggers, variants, and compliance.

  • Configurable cadence definitions per communication type
  • Practice-type variants (cosmetic, family, specialty)
  • HIPAA compliance enforcement (consent, opt-out, quiet hours)
  • A/B testing at network level

Message Generator

LLM-powered message composition with deep personalization — patient context, practice context, and network-optimized templates.

  • Merges patient, practice, and treatment data
  • Tone guidelines: warm, professional, never salesy
  • PHI validation before output
  • Multi-variant generation for A/B testing

Voice AI Agent

Natural conversational voice for inbound call answering and outbound reactivation calls with real-time PMS integration.

  • Practice-specific personality and knowledge
  • Real-time PMS schedule lookup during calls
  • Books appointments directly into PMS
  • Graceful handoff to human when out of scope

Network Learning Engine

Continuous optimization across all 260 practices. What works at one practice improves all of them.

  • Analyzes message performance across demographics and timing
  • Statistical significance requirements before promoting changes
  • Monthly optimization cycles with weekly trend monitoring
  • Practice-level guardrails and opt-outs

Graduation Manager

Manages the trust-building protocol — from Supervised to Monitored to Autonomous — independently per communication type.

  • 3 autonomy levels per communication type per practice
  • Graduation: <5% edit rate, >90% approval, 0 compliance flags
  • 30-day supervised period for new practices
  • HQ override capability
Decisions

Architectural Decision Records

Key architectural choices and the reasoning behind them.

ADR-1: Agent-Per-Practice vs. Shared Agent Pool

▼
Decision: One dedicated agent instance per practice.

Rationale: Dedicated agents maintain practice-specific context (providers, tone, patient history patterns) without context-switching overhead. At 260 practices, the cost is manageable. The agent IS the practice's communication personality — sharing it dilutes that. Aligns with Practice Concierge pattern.

Tradeoff: Higher infrastructure cost vs. better personalization and simpler debugging (one practice's issues don't affect others).

ADR-2: NexHealth as PMS Integration Layer

▼
Decision: Use NexHealth Synchronizer API as the universal PMS connector instead of building direct integrations.

Rationale: NexHealth supports 20+ major dental PMS systems, covering ~80%+ of SGA practices. One API instead of 20+. NexHealth handles PMS-specific quirks. For the remaining ~20% on unsupported systems, build lightweight adapters with manual event triggers.

Tradeoff: Vendor dependency + API cost at scale vs. years of custom integration work avoided.

ADR-3: Twilio for Communication Transport

▼
Decision: Use Twilio's HIPAA-eligible APIs for SMS, voice, and email instead of dental-specific vendor platforms.

Rationale: At 260 practices, annual savings of $440K-$785K vs. vendor platforms. More importantly, building on Twilio means SGA owns the intelligence layer. The playbook, network learning, and agent behavior are all proprietary IP. Vendor platforms mean renting someone else's intelligence.

Tradeoff: Higher development effort vs. lower ongoing cost + full IP ownership + no vendor lock-in.

ADR-4: Graduated Autonomy Protocol

▼
Decision: New practices start in Supervised mode and graduate to Autonomous over 30-60 days, rather than full auto from Day 1.

Rationale: Graduated autonomy solves adoption. Skeptical staff review messages before they send, building trust AND generating fine-tuning data. Once quality is proven (<5% edit rate), the practice graduates. This is how you onboard 260 practices without a training program.

Tradeoff: Slower initial ROI (30-day supervised period) vs. dramatically higher adoption rate and staff trust.

ADR-5: Network Learning as Core Architecture

▼
Decision: The playbook is dynamic — updated monthly based on network-wide outcome data, rather than static rules.

Rationale: 260 practices sending thousands of messages daily is a massive natural experiment. The network learning engine turns SGA's scale into a competitive moat. Over 12 months, the optimized playbook becomes SGA's most valuable communication IP — trained on SGA's specific patient population. No vendor can replicate this.

Tradeoff: More complex architecture + regression risk vs. continuous improvement and massive long-term competitive advantage.

Ecosystem

Integration with SGA Platform

The Comms Engine is the execution layer in a 3-sprint system: Practice Concierge (interface), Funnel OS (data), and Comms Engine (automation).

Practice Concierge

The Comms Engine agent IS the Practice Concierge agent, extended with communication capabilities. Same agent instance, new skill set. Practices interact with ONE agent — not separate systems for storefront management and patient communications.

Funnel OS

All communication events and outcomes feed the Funnel OS data layer. The executive dashboard shows comms performance alongside funnel metrics. The Practice Nerve Center weekly summary IS the Comms Engine weekly summary — same interface, same 20 minutes.

Convergence Vision

One agent per practice that manages storefront, communications, and operations. One data platform that tracks everything. One interface (20-min weekly summary) for practice staff. The three sprints converge into a single coherent platform.
