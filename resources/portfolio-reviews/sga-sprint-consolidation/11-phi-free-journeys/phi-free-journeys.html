Executive Summary

The business case for building PHI-free features first, then layering in patient-level capabilities.

8 wks
Faster to Working Software

The original 3-workstream platform design requires PHI across 21 data flows, of which 7 are high-risk (patient PII, appointment data, call transcripts with verbal PHI). Every one of these flows requires BAAs, HIPAA-compliant infrastructure, and compliance middleware before development can begin.

By deferring PHI, we can begin development immediately on approximately 65% of platform features. Phase A (No PHI) gets working software in front of users faster — enrollment pipelines, executive dashboards, the full Culture OS, and most of the Content Engine. Phase B (Add PHI) layers in patient-level features after BAAs, compliance middleware, and PHI filters are production-ready.

The result: working software at Week 10 instead of Week 18, dramatically lower HIPAA compliance risk during the build, and a more focused development effort in both phases.

Phase Plan

Two-phase approach: build the PHI-free foundation first, then layer in patient-level capabilities.

Phase A: PHI-Free Foundation

Weeks 1–14
  • Shared Infrastructure: Practice Data Hub (practice/staff data only), Auth, Kafka, Asset Registry
  • WS3 Full Build: H-Score Engine, Morning Huddle OS, CCA Academy, Champion Network, Recognition
  • WS2 Core: Studio Request Portal, Editorial Queue, Distribution Monitor, Brand Management
  • WS1 Foundation: Enrollment Pipeline, Storefront Gen, Executive Dashboard (aggregate KPIs), Portfolio Heatmap, HQ Ops tools
  • No BAAs needed yet — only processing practice, staff, and brand data
  • Synthetic/demo data for dashboard development during build

Phase B: PHI Layer

Weeks 15–24
  • Execute BAAs with NexHealth, Anthropic, OpenAI, Cloudinary
  • Build PHI Filter + Compliance Middleware
  • PMS Integration: real patient appointment data
  • Patient Communications: Twilio HIPAA tier
  • Concierge Agent, Automation Engine, Call Dashboard
  • Patient photo capture + de-identification pipeline
  • Patient-level funnel analytics
  • Voice AI pipeline

Risks & Mitigations

Key risks of the phased approach and how to address them.

Risk Impact Mitigation
Phase A delivers "half a product" Medium Design Phase A to be genuinely useful: enrollment, storefront, and culture tools are complete workflows. Users see dashboards that work with aggregate data — not empty shells.
PHI integration in Phase B is harder than building from scratch Medium Design Phase A architecture with PHI hooks (schema fields, API contracts) — just don't populate them yet. The data model anticipates PHI; the UI renders gracefully without it.
BAA procurement takes longer than Phase A Low Start BAA conversations in Week 1 regardless — they run in parallel. Phase B start date flexes if BAAs aren't done, but Phase A delivers value regardless.
WS1 team has less to build in Phase A Low WS1 team focuses on infrastructure quality, test coverage, and the enrollment/storefront/dashboard features that are available. They also build the PHI hooks that Phase B will need.
