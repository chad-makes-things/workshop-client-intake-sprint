Workshop Intake Engine

Client Onboarding Intake Sprint — Architecture Prototype
Sprint Date: 2026-03-07

Sprint Overview

Problem Statement

Pre-workshop client material collection is manual, fragmented, and unscalable. Materials arrive via text, email, Dropbox, and file shares. The facilitator manually collects, organizes, and maps these inputs into a structured prompt framework that generates a market narrative. This process takes 3-5 hours per client and scales linearly with client volume, creating a revenue ceiling.

Long-Term Goal

In 2 years, workshop clients arrive at Day 1 with a complete, high-quality market narrative already built -- by submitting their raw materials through a guided intake system that automatically collects, organizes, and synthesizes business context into the structured framework the workshop requires.

Chosen Solution

"The Interview" + "The Head Start" Hybrid -- A conversational guided intake flow with web-scraping pre-population. The system scrapes the client's web presence before the interview begins and pre-fills framework sections. The client reviews pre-populated content and fills gaps through guided questions, file uploads, and flexible input (text, links, voice).

Sprint Target

Guided Intake and Collection (Steps 2-6) -- Design the system that replaces scattered multi-channel collection with a guided, flexible intake experience that works for all client types. This is where the highest-risk behavioral assumptions live.

User Types

Client: Business representative (founder, marketing lead, ops) completing the guided intake. Varying digital sophistication.

Facilitator: Workshop business owner. Reviews structured submissions, approves follow-ups, triggers narrative generation. Intellectual architecture embedded in system; manual labor eliminated.

Sprint Questions

  • HIGH Can we get clients to actually submit materials through a structured system instead of texting, emailing, and sending files ad hoc?
  • HIGH Will the system produce a market narrative of equal or better quality than what the facilitator creates manually?
  • HIGH Can we handle the full spectrum of client digital sophistication?
  • MEDIUM Can we build this without making the pre-workshop experience feel impersonal or transactional?
  • MEDIUM Will the facilitator actually trust and adopt this system?

Component Summary

Intake Flow Engine

Client-facing guided interview. Typeform-style progressive disclosure with branching logic, pre-populated review mode, multi-modal input, and progress tracking. Serves Steps 2-8.

Facilitator Dashboard

Framework-organized view of all client submissions. Completeness indicators, source attribution, one-click follow-up approval, and narrative package compilation. Serves Steps 9-10.

AI Content Processor

Analyzes, categorizes, and synthesizes content from all sources against the prompt framework. Generates pre-populated drafts, detects gaps, drafts follow-ups. Serves Steps 1, 4-5, 9-10.

Session Manager

Persists session state with auto-save and magic-link tokens. Enables cross-device resume without login. Serves Step 7.

Web Scraping Engine

Crawls client website, LinkedIn, social profiles, and review sites on booking confirmation. Produces pre-population content. Serves Steps 1, 4.

File Processing Pipeline

Ingests uploaded files (PDF, DOCX, audio, images) and pasted links. Extracts text, transcribes audio, parses documents. Serves Steps 5-6.

Notification Service

Sends contextual communications via email and SMS. Welcome emails, pause reminders, submission alerts, follow-up delivery. Serves Steps 1, 7, 9.

Framework Template Store

Maintains the facilitator's prompt framework structure, question tree configuration, and mapping rules. Versioned and facilitator-editable. Serves all steps.

Storyboard

10-step user journey from workshop booking to framework-ready narrative package. Each step shows the scene, the user action, the system response, and the critical design detail.

1
The Welcome Email
Client (Sarah) sees an email from the facilitator: "Getting started on your workshop -- one link, 20 minutes." Web scraping has already run in the background since booking was confirmed two days ago.
Sarah opens the email. It contains warm, personal copy and a single button: "Start Your Workshop Prep." A note says: "We've already done some homework on your company."
The system crawled her website, LinkedIn, social profiles, and review sites. An AI model generated a first-pass draft mapped to framework categories. The email was held until the scrape completed.
Email comes from the facilitator's real name and address, not a system address. For thin-web-presence clients, copy adjusts to "We've pulled together some initial thoughts" to calibrate expectations.
2
Facilitator Video Kickoff
Sarah lands on a clean, branded page with a 60-90 second video of the facilitator speaking directly to camera.
Watches the video. Facilitator explains the interview, estimates 20 minutes, says "pause anytime." Sarah clicks "Let's Go."
Links client identity to pre-scraped data and framework template. Initializes progress tracker at 0%. Logs session start.
Video is recorded once and reused (not personalized). Must feel personal: direct eye contact, conversational tone. Page is minimal -- no sidebar, no login wall, no clutter.
3
Self-Segmentation
First screen: single question with 3-4 large, tappable options, Typeform-style.
Sarah reads "Which sounds most like your situation?" and selects "We have some materials but they're scattered or outdated."
Configures the interview branching path: moderate depth, pre-populated review + upload prompts for existing documents, balanced between review-mode and question-mode.
Options use client language, not framework jargon. No option implies inadequacy. System handles wrong-segment selection gracefully by adapting mid-flow.
4
Pre-Populated Review
Instead of a blank text field, Sarah sees a card with "Your Market" and a paragraph already filled in from the web scrape, lightly highlighted.
Reads the AI draft. Clicks "Close, but let me edit." Adjusts customer age range and adds a sentence about a new market segment not on the website.
Saves edits, tags section as "client-reviewed and modified." Logs the delta between AI draft and client's version (deltas are high-value signals). Progress bar advances.
Three response options always visible: "This is right" / "Close, let me edit" / "This is off, let me rewrite." For thin scrape results, system switches to question-mode seamlessly rather than displaying a thin draft.
5
Guided Deep-Dive
The interview reaches Competitive Landscape where the web scrape found little. System switches to guided questions.
Sarah types three competitor names. Gets smart follow-ups. Uploads a competitive matrix PDF when prompted in context.
Tags each answer to the framework slot. Processes uploaded PDF (extracts competitive claims). Merges typed answers with document content. Skips redundant follow-ups since the PDF covered them.
Upload prompts are contextual ("Upload your competitive analysis"), not generic. If client says "we don't have competitors," system pivots: "Who were your customers using before they found you?" File processing fast enough to adapt subsequent questions.
6
Flexible Input
Brand Story section. Sarah has strong feelings but has never written them down. The text field feels limiting.
Taps the microphone icon and speaks for 90 seconds about her founder story. Later, pastes a link to a podcast episode that captures her perspective.
Transcribes voice in near real-time. Queues podcast link for async processing. All inputs (text, voice, link) merge into unified Brand Story section. Notes that verbal storytelling was richer than typed answers.
Voice recording works on mobile and desktop with a single tap. All four input types (text, voice, file, link) available for every question. Link processing can happen async, but client sees confirmation that link was received.
7
Pause and Return
Sarah is 65% through. Her phone rings. She closes the browser tab without saving.
Leaves. Returns that evening on her phone by tapping the magic link from a reminder message. Lands exactly where she stopped.
Auto-saved after every answer. Sends reminder 4 hours later: "You're 65% done -- about 8 minutes left." Includes one-line summary of what was captured. Magic link works cross-device without login.
No "are you sure you want to leave?" dialogs. Max one reminder per 24 hours. Reminder tone adapts to workshop proximity (gentle when distant, direct when close). Tone is "we saved your progress," never "you haven't finished."
8
Review and Submit
Summary screen shows all framework sections with content: some pre-populated and confirmed, some edited, some from answers and uploads.
Scrolls through summary. Green checkmarks (complete) and yellow flags (thin, optional to add more). Adds two sentences to a flagged section. Taps "Submit."
Packages all inputs into structured framework document. Attaches metadata (AI-drafted vs. client-authored, source provenance). Calculates completeness score. Notifies facilitator: "87% completeness. 3 files processed. Review ready."
Summary is scannable, not overwhelming. Yellow flags are optional -- never block submission for incomplete sections. Completeness score is internal only; client never sees a grade.
9
Facilitator Dashboard
Facilitator opens the dashboard. Pipeline view shows four active clients. Clicks into Sarah's completed intake.
Views framework sections as tabs/cards with content type breakdown. Reviews Competitive Landscape (merged from typed answers + uploaded PDF). Flags thin "Market Sizing" section. Clicks "Request Follow-Up."
System drafts a targeted follow-up in the facilitator's voice. Facilitator approves with one click. Follow-up sent to Sarah. Dashboard updates to show pending follow-up.
Dashboard organized by framework section, not submission chronology. Content provenance visible but not intrusive. Follow-up messages drafted in facilitator's voice, not system-speak. Facilitator can edit before sending.
10
Framework-Ready Package
Three days later. Follow-up answered. Intake at 94% completeness. Workshop in 8 days.
Facilitator clicks "Generate Narrative Package." Reviews compiled prompt inputs with source attribution. Adjusts one section. Triggers narrative generation.
Compiles all framework sections into structured prompt inputs for the existing AI narrative pipeline. Each slot filled with source citations ("from client voice recording," "from uploaded competitive matrix, page 3"). Facilitator's manual effort: ~15 minutes. Previous: 3-5 hours.
Output maps one-to-one to the facilitator's existing prompt framework -- the system adapts to the facilitator's workflow, not the reverse. Source attribution is essential for confidence. This step bridges the intake system (this sprint) and the narrative generation system (future sprint).

Architecture Decision Records

Key architectural decisions, the alternatives considered, and the rationale for each choice.

ADR-1: Branching Logic Architecture
Context

The intake flow must adapt based on client self-segmentation (4 paths) and prior answers within each path. Questions, depth, and input prompts change based on selections. This branching logic is the intellectual core of the system.

Decision

Store the question tree as a JSON configuration in the Framework Template Store, not as hardcoded application logic. The tree is a directed graph where each node is a question/prompt, edges are conditional transitions, and nodes reference framework section keys. The Intake Flow Engine traverses this graph at runtime.

Alternatives Rejected

Hardcoded in application code -- rejected because the facilitator must adjust question sequences without code deployments. The question tree is a content artifact, not an engineering artifact.

Third-party form builder (Typeform/Tally) -- rejected because the pre-populated review mode (confirm/edit/rewrite of AI-drafted content) is not supported by any commercial form tool.

Rule engine (Drools) -- rejected as over-engineered. A JSON graph with a lightweight traversal engine is sufficient for 4 segmentation paths with 20-40 questions each.

Consequences

Facilitator can modify the question tree through configuration. New framework sections can be added without application changes. Requires a dedicated "question tree validator" to verify all paths are reachable and produce complete framework coverage.

ADR-2: Web Scraping Strategy
Context

The system must scrape client websites before the intake begins to pre-populate framework sections. Websites vary enormously: static HTML, JavaScript SPAs, sites behind login walls, minimal single-page sites.

Decision

Use a two-tier scraping approach. Tier 1: HTTP-based scraping with cheerio/jsdom for static pages. Tier 2 (fallback): headless browser (Playwright) for JavaScript-rendered sites. Limit crawl depth to 2 levels from homepage. Priority pages (about, services, team, blog) crawled first. Cap at 25 pages per site. LinkedIn and social profiles scraped via separate rate-limited workers.

Alternatives Rejected

Full recursive crawl -- large corporate sites could produce thousands of irrelevant pages.

Third-party scraping API (Diffbot) -- viable fallback but adds cost and vendor dependency for a core capability.

Client-submitted URL list -- adds friction and defeats the "we did our homework" value proposition.

Consequences

Scrape must complete in under 5 minutes (before welcome email sends). Pages behind login walls are skipped and flagged. Thin-web-presence clients trigger adjusted email copy. A failed scrape does not block the intake flow.

ADR-3: Multi-Modal Input Handling
Context

Clients must provide input via text, voice recording, file upload, and link pasting for any question. Each type requires different processing.

Decision

Every question supports all four input types simultaneously. Text is the default with voice, file, and link as secondary options. All non-text inputs processed asynchronously -- client receives immediate confirmation and can proceed without waiting. Processed content merged into framework sections retroactively.

Alternatives Rejected

Input type per question -- removes client agency. A founder who prefers speaking should be able to voice-record any answer.

Synchronous processing -- processing latency (10-30 seconds) would stall the conversational flow.

Post-submission upload -- contextual prompts within the flow produce higher-quality, more targeted uploads.

Consequences

System handles eventual consistency -- compiled sections may update after the client has moved past them. Dashboard shows processing status. Review/submit screen reflects most current state including recently processed items.

ADR-4: Session Management and Cross-Device Resumption
Context

Clients will start on one device, close the browser, and return hours or days later on a different device. Must preserve exact position without account creation.

Decision

Use magic-link token-based sessions. Unique session token embedded in URL and stored in HTTP-only cookie. Session URL included in all reminders. Auto-save fires after every interaction. Tokens expire 30 days after workshop date. State stored server-side (PostgreSQL + Redis cache).

Alternatives Rejected

Account creation with email/password -- disproportionate friction for a one-time, 20-minute flow.

Email-based OTP on each return -- adds a step on every resume. Magic links achieve the same security without extra friction.

Browser-only session (cookies/localStorage) -- does not support cross-device resumption.

Consequences

Magic links must be treated as sensitive (single-use per click, token refreshes on access). Must handle concurrent access gracefully. No client-side session storage dependency.

ADR-5: Framework Mapping Approach
Context

All client inputs must be mapped to the facilitator's prompt framework categories. This mapping bridges raw input and structured output.

Decision

Use a hybrid mapping approach. First-pass: structural -- each question is pre-tagged with its target framework section (handles ~70-80% of content with 100% confidence). Second-pass: AI-driven -- uploaded files, voice recordings, and links analyzed by LLM and distributed across relevant sections with confidence scores. Facilitator can override AI mapping in the dashboard.

Alternatives Rejected

Purely structural mapping -- uploaded files span multiple sections. Single-section tagging loses information.

Purely AI-driven mapping -- introduces unnecessary uncertainty for content with an obvious home.

Client-directed mapping -- exposes framework structure to the client, who should never need to understand it.

Consequences

Most content mapped with certainty via structural tagging. Only multi-section content requires AI classification, reducing cost and error rate. Dashboard shows provenance and mapping confidence. Initial calibration sessions with the facilitator are essential for AI mapping prompts.
