System Components
Eight components, each with a clear responsibility and defined interface. Together they form the complete Studio Automation system.
Component 1

Intake Channel Adapters

Normalize inbound messages from Teams, Slack, and email into a standard request object. Practice identified by sender email. Attachments (logos, photos) auto-upload to the practice's Canva brand kit.

Bot Framework SDK Slack Bolt SDK M365 Email API
Component 2

Studio Agent Core

The intelligence layer. Understands intent, runs conversational intake (max 3 questions), fires the medium challenge, assembles the brief, and routes to automated generation or human designer queue.

Claude API Intent Classification Medium Challenge Logic
Component 3

Practice Profile + DAM

Per-practice brand kits in Canva Enterprise: logo, colors, fonts, approved photography. Brand health score (0-100) flags practices below 40 for asset lift before automation can serve them.

Canva Enterprise Brand Health Score 160 Brand Kits
Component 4

Design Generation Engine

12 brand-locked templates covering ~85% of request volume. Canva API populates variable fields (offer text, CTA, dates, photo) from the brief. Preview images generated in ~45 seconds.

Canva API 12 Templates Variable Population
Component 5

Approval Workflow

Practice preview via original channel. Text-only revision in <2 minutes. CD mobile push notification with one-tap approve/reject. Practice confirms content before CD reviews brand quality — not after.

60-sec CD Review Mobile Push 2-Hour SLA
Component 6

Digital Deployment

Routes approved digital assets to patient communication platform (Weave), in-office display system (TV slides), and social scheduling tool. Phase 1: email + TV slide. Social and full TV automation in Phase 2.

Weave API TV Display Queue Social Scheduler
Component 7

Print Fulfillment

Nominated vendor API (Gelato) receives approved print-ready files with pre-configured specs. One-click order for practice managers. Studio never touches individual print orders. Tracking goes directly to the practice.

Gelato API Auto-Spec Direct-to-Practice
Component 8

Studio Analytics Dashboard

Request volume by channel, medium split trend, turnaround tracking, brand kit health per practice, CD review time, and 90-day inactive practice flag for proactive outreach. Studio's visibility layer.

Architectural Decision Records
Seven decisions that shaped the architecture — with the reasoning that makes them defensible.
ADR 1
Email as the third intake channel (studio@sgadental.com)
Teams and Slack reach Gen4 practices well — but legacy SGA and MODIS practices may not be on those platforms. Email is the universal access layer. Every practice manager has it, regardless of which collaboration tool the practice uses. It also creates a searchable audit trail of every request without additional infrastructure. A dedicated studio email address is also legible to practice managers as a "real thing" — more so than a bot mention in a channel they don't regularly use.
ADR 2
Claude API for the agent core, not a decision-tree chatbot
A decision-tree chatbot would handle "I need a business card" correctly. It breaks on "We're doing a Lunch & Learn at the perio office next week and need some collateral." The dental practice context is complex — the agent needs to understand what a liner special is, what a PRD does, why a doctor referral card is different from a patient-facing flyer. Claude's language understanding handles the edge cases that would require hundreds of decision tree branches to cover manually. The medium-challenge logic is also a nuanced conversation, not a binary yes/no — it requires understanding the practice's specific situation.
ADR 3
Canva Enterprise as both DAM and design tool (not separate systems)
Using a dedicated DAM (Brandfolder, Bynder) plus a separate design tool (Adobe Creative Cloud, Figma) adds a file-transfer step, two vendor relationships, two pricing models, and a more complex API integration. Canva Enterprise handles both functions with a single API. The Brand Kit feature provides per-practice asset storage with brand locking. The Connect API allows programmatic template variable population and export. The trade-off is that Canva's DAM capabilities are less sophisticated than dedicated DAM tools — but for 160 practices with relatively simple asset sets (logo, colors, photos), Canva's capabilities are sufficient.
ADR 4
Template library starts at 12 — studio-admin expandable, not engineering-gated
12 templates covering business cards, appointment reminders, flyers, TV slides, social posts, referral cards, review cards, and welcome packets handle approximately 85% of request volume based on the expert conversation. The critical design decision: adding a new template is a studio designer task in Canva — not an engineering sprint. This removes the bottleneck that would otherwise slow template expansion. The studio team can identify gaps from analytics (what's being escalated to human designers?) and add templates without a software release cycle.
ADR 5
Practice preview before CD review (not after)
The standard workflow in most agencies is: design → client review → revisions → internal sign-off. We reversed the middle two steps. If the CD approves an asset and then the practice requests a content change, the CD must re-review the revised version. By having the practice confirm content accuracy first, the CD's review is exclusively a brand quality judgment on a finalized asset. This reduces the probability of a post-CD change request from approximately 30% to under 5%, and keeps the CD's cognitive load focused on brand — not proofreading offer text.
ADR 6
Digital deployment integrations are phased — TV slide manual in Phase 1
The variety of in-office display systems across 160 practices (Apple TV + a management app, dental-specific display software, simple HDMI sticks) makes a single-API TV integration impossible in Phase 1. Rather than block the launch, Phase 1 treats TV slides as a file delivery — the agent sends the asset, the practice manager uploads it. Phase 2 maps the most common display systems in the Gen4 network and builds the integration for those. The patient email integration (Weave) is simpler and goes live in Phase 1. This approach ships value in Phase 1 without waiting for perfect automation.
ADR 7
Nominated vendor enforced through workflow — not policy
Policy-based vendor consolidation ("you should use this vendor") fails because practice managers under time pressure will use whatever is fastest — Staples, a local print shop, the office color printer. Workflow-based enforcement works because the only "Order Print" button in the system routes to Gelato. Practices can download the print-ready PDF and use any vendor they want — but they get no studio support for that path, no correct file specs, and no tracking. The nominated vendor path is so much easier that practices choose it without being told to. Practices that insist on local vendors opt themselves out of the workflow, which is acceptable.
